CLIENT SIDE PREDICTION NETWORKED PHYSICS SYSTEM

First of all, Vandalhalla has a Client Predicted, Server Authoritative networked physics system, a system that innovate the FPS industry.

What does it mean ?

In a standard multiplayer game that uses physics as a gameplay mechanic. It's the SERVER instance that dictate how an object interact physically with another object.
How does it work ?
How does it work ?
Standard Network

For example in a standard network architecture. If a player use a car,
when he presses the forward key first, the key press is sent to the SERVER,
than the SERVER process it and make the car move forward,
and at last this movement is sent back to all players.

Let's say all that takes 200ms (a 1/5 of a second).

Client Predicted Network

In a CLIENT side predicted physics system,
when the player presses the forward key, the car starts to move directly on the CLIENT and the key is send to the SERVER,
Then the SERVER process the key, and it creates the car movement as well in the SERVER instance (like in a standard architecture),
and so the movement is sent back to all players.

BUT there its own more step, the "correction step". As the CLIENT who pressed the key in the first place, we are now correcting the movement of the car based on the movement send by the SERVER and our current movement.

This has a consequence to slow a little bit our car to match the SERVER instance reality on the CLIENT instance.
The travel time between the key press and SERVER movement being received by us is still 200ms.
But this time we did not had to wait this delay to see our car movement.
Why we need that system ?
Why we need that system ?
I wanted to integrate such system into Vandalhalla based of the concept of Game fell from Steve Swink, mainly because of one principal "Realtime controls" which stand for having responsive controls in game.

That mean that I needed a networking system based on physics that respond in less than 200ms.

In addition, now if we apply all this to a FPS, we can see our arrows being sent directly after the mouse button was press or even rocket jump without any delay. Which improve directly the game feel and the precision of the movements.

What about cheating ?
What about cheating ?
And if you are worrying about cheating, there is no way for a cheater to get a faster car.

Because the cheater is only sending the key press to the SERVER, this means that even if the car go faster on the cheater CLIENT, the SERVER has the right speed, and so he will always send back the real position of the cheater car.

And so as a consequence, the cheater will only see his car being teleported back to the SERVER position because on the correction step.
As mentioned before, the correction will adjust the speed of the car to match the SERVER reality, but if the results are too different between the CLIENT and the SERVER.
The game will set the cheater car position to the SERVER position of the car.
And other players of the same race will never see the teleportation of the cheater because they never had been sent the cheater wrong position in the first place.
